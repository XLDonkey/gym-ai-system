What Is GitHub?

A plain-English explanation — no coding experience required.

If you've never heard of GitHub before, here's the simplest way to think about it:

💡
The One-Line Explanation

GitHub is like Google Drive for code — but smarter. Every change is saved with a note explaining what changed and why. You can go back to any previous version. Multiple people can work on the project without overwriting each other. The entire project history is permanently preserved.

You already know how Google Drive works: you create folders, put files in them, and the files are saved online so anyone with the link can see them. GitHub works the same way — but it's built specifically for code projects, and it has some extra superpowers that Google Drive doesn't have.

The Key Superpower: Nothing Is Ever Lost

In Google Drive, if you accidentally delete a file or overwrite it with bad changes, recovering it can be tricky. In GitHub, every single change ever made is permanently recorded. You could go back to exactly what the project looked like on the day it was created. This is what developers mean when they say "version control" — GitHub controls the versions.

The Six Concepts You Need to Know

There are six terms that come up constantly when talking about GitHub. Here's what they all mean:

📁
Repository

Repository (Repo)

The main project folder. It contains all the code, documents, and files for a project. Our AI Overseer project lives in one repository.

Like a Google Drive folder for the whole project — but it remembers every version of every file, forever.
💾
Commit

Commit

A saved snapshot of the project at a specific moment in time. Every commit has a message explaining what changed and why.

Like hitting "Save" in a Word document — but every save is kept forever, labelled with a note, and you can go back to any of them.
🌿
Branch

Branch

A parallel copy of the project where you can make changes safely. We have a "main" branch (always working) and a "dev" branch (work in progress).

Like photocopying a document before editing it — you work on the copy and decide later whether to replace the original.
⬆️
Push

Push

Uploading your saved changes from your computer to GitHub. Once you push, the changes are online and anyone with access can see them.

Like syncing files to Google Drive. Your changes go from your computer to the cloud.
🌐
GitHub Pages

GitHub Pages

A free feature that turns files in your repository into a live website. Any HTML file becomes accessible at a public URL — no server needed, no hosting fees.

Like Google Drive's "Publish to web" feature, but for proper websites. This is how our demo runs live.
📄
README

README

A special file that serves as the front page of a repository. When someone opens our project on GitHub, the README is the first thing they read.

Like the description on the front of a folder — it tells you what's inside before you open it.

The Branch Strategy

How we keep the live demo always working while development happens in parallel.

A branch is like making a photocopy of the entire project to work on. You can edit the copy freely without affecting the original. When you're happy with your changes, you "merge" the copy back into the original. If the copy turns out to be bad, you throw it away — the original is untouched.

We use a specific branching strategy to ensure the demo is always working, even when Donkey is in the middle of a big change:

Branch Structure — Live View
main
✅ always live · always working
↓ branched from main
dev
🔧 work in progress
↓ feature branches off dev
feature/lstm-model
→ merged to dev when done
feature/face-id
→ merged to dev when done
feature/weight
→ in progress

The Three Types of Branch

🟢
main

Main Branch

The live, always-working version of the project. The GitHub Pages demo runs from this branch. Donkey never makes experimental changes directly here. Only tested, confirmed-working code gets merged into main.

Like the master copy of a document that's being actively shown to clients. You don't scribble drafts on it — you work on a photocopy first.
🟣
dev

Dev Branch

Where Donkey actually builds things. This could be broken at any time during development — that's fine here. Once a feature is complete and tested on dev, it gets merged into main.

Like the workshop. Messy, things get taken apart, half-built stuff everywhere — but that's what workshops are for. The showroom (main) stays tidy.
🟡
feature/...

Feature Branches

Created for a single specific piece of work — e.g., building the LSTM model, adding face ID, or integrating weight detection. These branch off dev, get developed in isolation, then merge back to dev when complete.

Like a separate notepad for each task. Work on one thing without getting mixed up with everything else.
🔁 The Golden Rule

Build on dev → test it → confirm it works → merge to main → the demo URL automatically updates within 2 minutes.

Why Does This Matter?

Without this strategy, there would be a real risk of breaking the demo for Matt or a stakeholder at exactly the wrong moment — like just before an investor meeting. The branch strategy means:

  • The demo URL is always reliable — it only ever shows tested, working code
  • Donkey can work on complex features that take days or weeks without disrupting anyone
  • If something goes wrong during development, it's on dev — not visible to anyone
  • The frozen demo/ folder adds an extra layer of safety for presentations

What Happens When Something Breaks

Why nothing is ever permanently lost in GitHub — and what to do if the demo looks wrong.

🛡️
Nothing is ever permanently lost in GitHub

This is the single most important thing to understand. No matter what goes wrong during development, the entire history of the project is preserved. Every version that ever worked is still there, accessible in under 2 minutes.

What Happens If Donkey Breaks the Demo

  1. 1

    The commit history shows exactly what changed

    Every change is recorded with a timestamp and a message. If something breaks after a commit, it's immediately obvious which change caused it. There's no guessing.

  2. 2

    Donkey can revert to any previous version in under 2 minutes

    Using the commit hash of the last working version, Donkey can restore the entire project to that exact state with a single command. The broken change is effectively undone — like hitting Undo, but for the entire project.

  3. 3

    The demo URL goes back to working as if nothing happened

    After the revert is pushed to GitHub, GitHub Pages republishes the restored version. Within 2 minutes, the URL is working again. No lasting damage.

The Two Layers of Protection

🌿

Layer 1: The Branch Strategy

Donkey only merges tested code into main. The demo URL (pose/alpha.html) only ever runs code that has been tested on the dev branch first. Experimental or broken code never reaches the live URL under normal conditions.

🧊

Layer 2: The Frozen Demo

The demo/alpha.html file is never modified during development. It is a deliberately frozen, known-good version kept purely for presentations. Even if the main demo had issues, the frozen demo always works.

Repository (Repo) A project folder containing all the code, documents, and files for a project. Like a Google Drive folder, but with complete version history. Our project lives at github.com/XLDonkey/gym-ai-system.

Branch A parallel copy of the project where changes can be made safely without affecting the original. We have main (live), dev (work in progress), and feature branches (isolated work on one thing).
Main Branch The live, always-working version of the project. GitHub Pages publishes from here. Only tested, confirmed-working code gets merged into main. Never broken intentionally.
Dev Branch Where Donkey builds things. Can be broken during development — that's fine here. Once a feature works on dev, it gets merged into main and the demo updates.
Feature Branch A branch created for one specific piece of work — e.g. feature/lstm-model. Keeps that work isolated from everything else until it's ready to merge.
Merge Combining changes from one branch into another. When work on dev is done, it gets merged into main. Like deciding to replace the original document with the improved photocopy you've been editing.
GitHub Pages A free GitHub feature that automatically publishes HTML files from a repository as a live website. No server needed. Our entire demo and documentation suite is hosted this way at xldonkey.github.io.
README A special file (README.md) that GitHub automatically displays on the repository's front page. It's the first thing anyone sees when they open the project — a description and introduction to the project.
Clone Downloading a full copy of a repository to your own computer, including the entire history. This is what developers do before starting to work on a project. Matt doesn't need to do this.
Fork Creating your own copy of someone else's repository under your own GitHub account. Used when you want to make changes to a project you don't have write access to. Not something we do in our current workflow.
Pull Request (PR) A formal request to merge changes from one branch into another. Used by developers to say "I've finished this work — please review it before adding it to the main project." Donkey reviews PRs from any future team members.
.gitignore A special file that tells Git which files should never be committed to the repository. We use it to exclude API keys (passwords), large video files, and other sensitive or oversized items. Security essential.
Version Control The general concept of tracking every change to files over time. GitHub is a platform for version control. "Using version control" means you're keeping a complete history of every change — nothing is ever permanently lost.
Revert Undoing a commit by creating a new commit that reverses it. If a change breaks something, Donkey can revert it — restoring the project to how it was before. Takes under 2 minutes.
Deploy Making a new version of the system available to users. For us, "deploying" means pushing to the main branch — GitHub Pages handles the rest automatically. No separate deployment step needed.
PWA (Progressive Web App) A website that behaves like a native phone app. Can be installed from the browser, works offline, and gets an icon on the home screen. Our frozen demo is a PWA — that's why you can add it to your tablet's home screen.
Service Worker A background script that makes PWAs work offline. Once the demo loads once, the service worker caches it so it can run even without internet. Critical for gym presentations where Wi-Fi may be unreliable.
JSON A file format for structured data — like a very precise spreadsheet. Our machine configs and model weights are stored as JSON files. You don't need to read them, but you'll see the .json extension in the file tree.
Markdown (.md) A simple text format where special characters create formatting. # Heading becomes a big heading. **bold** becomes bold. GitHub automatically renders Markdown files as formatted text — the README is a Markdown file.

Troubleshooting — When Things Go Wrong

Every common problem, its causes, and exactly how to fix it. Keep this section bookmarked.

Things go wrong. That's normal. The key is knowing exactly what to check and exactly what to do. Here are every common problem someone might encounter working with our GitHub setup.

🔴 "The demo URL isn't showing my changes"

CAUSE 1: Browser Cache

Your browser is showing the old cached version, not the new one. This is the most common cause by far.

Mac: Cmd+Shift+R    Windows/PC: Ctrl+Shift+R

This is a "hard refresh" — forces the browser to re-download everything, ignoring cache. Try this first, every time.

CAUSE 2: GitHub Pages Rebuilding

GitHub Pages takes up to 2 minutes to rebuild after a push. If you just pushed, wait 2 minutes and hard-refresh. Check the Actions tab on GitHub to see rebuild status — a green ✓ means it's done.

CAUSE 3: Changes Are on Dev, Not Main

GitHub Pages only serves from main. If you pushed to dev but haven't merged to main, the demo won't have your changes. Run git checkout main && git merge dev && git push origin main to deploy.

🔴 "I can't push my changes"

CAUSE 1: Not Authenticated

Git doesn't know who you are or doesn't have permission. Two options:

  • SSH key: run ssh-keygen -t ed25519, add the public key to GitHub Settings → SSH Keys
  • Personal Access Token: generate at GitHub Settings → Developer Settings → Tokens, use it as your password
CAUSE 2: Remote Has New Commits You Don't Have

Someone else pushed to the same branch after you last pulled. You need to pull their changes first, then push yours.

git pull origin dev     # get latest changes

Then push again: git push origin dev

CAUSE 3: Trying to Push Directly to Main

Main is protected. Direct pushes to main may be blocked. This is by design. Work on dev or a feature branch, then merge to main following the proper process.

🔴 "The demo was working and now it's broken"

Don't panic. This is fixable within minutes. Follow this exact process:

  1. 1

    Check the Last 3 Commits on Main

    Go to GitHub → gym-ai-system → the clock icon → commits on main. Which commit is the newest? What did it change? That's almost certainly the cause.

  2. 2

    Identify Which Commit Broke It

    In the terminal: git log --oneline shows a compact list of recent commits with their hash and message. If there were 3 commits today, one of them caused the break.

  3. 3

    Revert That Commit

    # Replace abc1234 with the actual commit hash from git log
git revert abc1234
# This creates a NEW commit that undoes the bad commit.
# The bad commit stays in history — git revert never deletes history.
# It just applies the opposite changes.
  4. 4

    Push the Revert to Main

    Run git push origin main. GitHub Pages rebuilds. Within 60–120 seconds, the demo is back to working. Total recovery time: under 5 minutes.

🟡 "I accidentally committed to main"

Not catastrophic — especially if the commit is small. You have two options:

OPTION A: Just Fix It on Main

If it's a small change that worked fine, just commit the follow-up fix directly to main too. Not ideal process, but no harm done. Learn for next time.

OPTION B: Revert and Redo Properly

Run git revert [hash] on main to undo the commit. Then redo the work on dev, merge to main correctly. This is the "right" approach if the change was substantial.

💡
Prevention is better than fixing

Before running git push, always run git branch to confirm which branch you're on. If it shows * main, switch to dev first: git checkout dev.

🟡 "The GitHub Pages Site Shows a 404"

CAUSE 1: Wrong File Path

The URL is case-sensitive. Alpha.html and alpha.html are different files. Check the spelling and case of the URL path against the actual filename in the repo.

CAUSE 2: GitHub Pages Not Enabled

Go to GitHub → Settings → Pages. Confirm it's enabled and set to deploy from main branch. If it got accidentally disabled, re-enable it — the site will be back up within 2 minutes.

CAUSE 3: Repo Made Private

GitHub Pages on a free account requires the repo to be public. If the repo was accidentally made private, go to Settings → Danger Zone → Change visibility → Make public. The site will come back automatically.

🟡 "I Don't Know What Changed Between Two Versions"

Two ways to compare any two points in time:

On GitHub (Visual):

Go to the commits list → click any commit → see a diff view. Green lines = added. Red lines = removed. Every changed file is listed. You can see exactly what Donkey typed.

In Terminal (git diff):
# Compare two specific commits by their hashes
git diff abc1234 def5678
# Shows every line that changed between those two moments.
# Green (+) = added, Red (-) = removed.

# Compare current state to a specific commit
git diff abc1234 HEAD
# HEAD = the latest commit on your current branch

# Compare two branches
git diff main dev
# Shows what dev has that main doesn't

🔵 Quick Reference — Emergency Commands

Emergency Git Cheatsheet
# Which branch am I on?
git branch

# What has changed since my last commit?
git status

# Show recent commits (compact)
git log --oneline -10

# Undo the last commit (keeps changes as uncommitted)
git reset --soft HEAD~1

# Revert a specific commit (safe — adds a new commit, doesn't delete history)
git revert [commit-hash]

# Get the latest changes from GitHub
git pull origin dev

# Discard ALL local changes (dangerous — use with care)
git checkout -- .

# Switch branch without losing uncommitted work
git stash           # save current work temporarily
git checkout dev    # switch branch
git stash pop       # restore saved work
🛡️
The Safety Net You Always Have

Git never permanently deletes committed work. Even if you run a command that seems destructive, the history is almost always recoverable. The only truly irreversible action in Git is force-pushing to delete remote history — which no one on this project should ever do. If you're unsure, ask before running any command with --force in it.
