Section 01 — The Vision

What We're Building

The XL Fitness AI Overseer is a live intelligence layer overlaid onto the physical gym floor. A network of ceiling-mounted 4K cameras feeds a real-time AI pipeline that tracks every member, counts every rep, and logs every set — automatically, with no phones, no QR codes, and no manual input.

The goal: full visibility over every machine, every person, every rep — in real time from above.
The End State

A member walks in, does their workout, and walks out. That evening they open the app and see a complete log of every exercise, every set, every rep, and every weight — automatically captured by the system. Zero effort. Zero friction.

Section 02 — The Problem

The Core Problem

Gyms are data black holes. Members walk in, lift weights, and leave. The only way to track progress today is to manually enter data into an app — which breaks the flow of the workout. Over 90% of gym-goers abandon workout tracking within the first month.

4. Try to remember the weight you just used

8. Give up on tracking after 3 weeks

For gym owners, the problem is even worse. They have no data on which machines are popular, when bottlenecks occur, how long members spend on each machine, or who is at risk of churning. Every business decision is made on gut feel.

The Opportunity

The gym generates thousands of data points every hour. We just need to capture them. The cameras are already there for security — we just need to make them intelligent.

Section 05 — Hardware

Camera Mounting Frame

We cannot mount cameras directly to the ceiling. Instead, we are building a suspended steel grid frame — a "lighting truss" or "camera rig" — that hangs below the ceiling. The cameras attach to this frame, giving us total flexibility in positioning.

A suspended camera rig. We build a grid of steel Unistrut or aluminium truss, suspend it from the main ceiling beams, and mount the cameras to the grid. This allows us to place a camera exactly where we need it, even if there is no ceiling beam directly above.

Why We Use a Frame

If you mount directly to the ceiling, you are restricted by where the structural beams are. If a beam isn't directly over the squat rack, your camera angle is ruined. By building a suspended grid frame (like a VR soundstage or a theatre lighting rig), we decouple the camera positions from the building's architecture.

  • Perfect Angles: We can slide the camera along the rail to get the exact top-down or 45-degree angle required.
  • Cable Management: The Cat6 cables run cleanly inside the channels of the frame, hiding them from view.
  • Modularity: When a new machine is added to the gym, we just unbolt the camera and slide it down the rail to the new position.
  • No Ceiling Damage: We only need a few anchor points to hang the entire frame, rather than drilling 20 separate holes for 20 cameras.
The frame system uses crossbars. Cameras attach via heavy-duty clamps (like Super Clamps or Unistrut brackets). This is the exact setup used in professional motion capture studios.

Section 07 — Hardware

Video Streaming Pipeline

Getting video from the camera into the AI model is not as simple as plugging in a USB webcam. IP cameras speak RTSP (Real-Time Streaming Protocol), a network protocol designed for low-latency video delivery. Understanding this pipeline is critical to building a stable system.

RTSP: the camera acts as a server, and our AI software acts as a client pulling the stream on demand.

Step 1: The RTSP URL

Every IP camera exposes one or more RTSP streams. The URL format is standardised across manufacturers. The camera streams H.265 compressed video, which must be decoded before the AI can process it.

rtsp-url-format.py
# Hikvision RTSP URL format:
rtsp://[username]:[password]@[camera_ip]:554/Streaming/Channels/101

# Example for Camera 01:
rtsp://admin:XLFitness2024@192.168.10.101:554/Streaming/Channels/101

# /101 = Main stream (4K, 30fps, H.265)  -- used for AI
# /102 = Sub stream (1080p, 15fps, H.264) -- used for face ID

# Test the stream with FFprobe:
ffprobe -v quiet -print_format json -show_streams \
  "rtsp://admin:pass@192.168.10.101:554/Streaming/Channels/101"

Step 2: Hardware-Accelerated Decoding

Decoding 4K H.265 video on the CPU is expensive — a single stream can consume 30-40% of a CPU core. With 20 cameras, that's impossible. Instead, we use the GPU's dedicated video decoder (NVDEC on NVIDIA, VideoToolbox on Apple Silicon) to decode all streams in parallel at near-zero CPU cost.

camera_capture.py
import cv2
import threading
import queue

class CameraStream:
    def __init__(self, camera_id, rtsp_url):
        self.camera_id = camera_id
        self.frame_queue = queue.Queue(maxsize=2)  # Small buffer = low latency
        
        # Use FFMPEG backend with hardware decoding
        self.cap = cv2.VideoCapture(rtsp_url, cv2.CAP_FFMPEG)
        self.cap.set(cv2.CAP_PROP_BUFFERSIZE, 1)
        
        self.thread = threading.Thread(target=self._capture_loop, daemon=True)
        self.thread.start()
    
    def _capture_loop(self):
        while True:
            ret, frame = self.cap.read()
            if ret:
                # Drop old frame if queue is full (prefer fresh frames)
                if self.frame_queue.full():
                    self.frame_queue.get_nowait()
                self.frame_queue.put(frame)
    
    def get_frame(self):
        return self.frame_queue.get(timeout=1.0)

Step 3: Multi-Camera GPU Batching

We do not process cameras one by one. Each camera has a dedicated capture thread that reads frames and pushes them into a thread-safe queue. A GPU inference worker pulls frames from multiple queues and processes them in a batch — dramatically more efficient than processing one frame at a time.

Section 09 — Software

The Full Software Stack

The software is not a single monolithic application. It is a collection of specialised services, each doing one job extremely well, communicating through message queues. Click each component below to learn more about its role.

GStreamer is a multimedia framework that handles the low-level work of connecting to RTSP streams, decoding H.265 video using the GPU's hardware decoder (NVDEC or VideoToolbox), and delivering raw frames to the AI pipeline. It is written in C and is significantly more efficient than pure Python alternatives.

FFmpeg is used as a fallback and for any video processing tasks (resizing, format conversion). OpenCV's VideoCapture uses FFmpeg under the hood when you pass a network URL. For Phase 1 (Mac), we use VideoToolbox hardware decoding. For Phase 2+ (NVIDIA), we use NVDEC via CUDA.

PyTorch is the primary deep learning framework. All models (YOLO, MoveNet, LSTM, FaceNet) are loaded and run through PyTorch. It provides automatic GPU acceleration via CUDA (NVIDIA) or MPS (Apple Silicon).

ONNX Runtime is used for production deployment. We export trained models to the ONNX format, which allows them to run faster with hardware-specific optimisations (TensorRT on NVIDIA, CoreML on Apple).

TensorRT (NVIDIA only) is the final step for maximum performance. It compiles the ONNX model into a highly optimised engine for a specific GPU, squeezing out every last millisecond of latency.

Redis serves two roles. First, it acts as a high-speed in-memory message queue — camera capture threads push frame references into Redis lists, and AI worker threads pop them off. This decouples the I/O from the compute and prevents any single slow component from blocking the others.

Second, Redis stores the live state of the gym — which person IDs are currently active, which machine they are at, their current rep count, and their last known face match. This state is updated every frame and queried by the dashboard in real time.

PostgreSQL is the permanent relational database. Every completed set is written here: member ID, machine ID, timestamp, rep count, estimated weight, and confidence score. This is the source of truth for the member-facing app and all analytics.

We use a write-ahead log (WAL) configuration for durability, and a read replica for the analytics dashboard to avoid query load on the write path. TimescaleDB (a PostgreSQL extension) adds time-series optimisations for workout history queries.

FastAPI is a modern Python web framework that provides the REST API consumed by the member-facing mobile app and the gym owner dashboard. It is asynchronous by default, which means it can handle hundreds of concurrent requests without blocking.

Key endpoints: /workouts/{member_id} (get workout history), /live/{gym_id} (get current gym state via WebSocket), and /members/register (enrol a new member's face).

YOLO Detection

The first job of the AI is to find the people. We use YOLOv8 (You Only Look Once, version 8) by Ultralytics — the fastest and most accurate real-time object detection model available.

YOLO detecting people in a busy scene — each person gets a bounding box with a confidence score in milliseconds.

YOLO processes the entire image in a single forward pass through the neural network. Unlike older two-stage detectors that first propose regions then classify them, YOLO does both simultaneously — hence "You Only Look Once". This makes it dramatically faster.

yolo_inference.py
from ultralytics import YOLO
import torch

# Load model (downloads automatically on first run)
model = YOLO('yolov8n.pt')  # 'n' = nano (fastest)
model = model.to('cuda')    # Move to GPU

results = model.predict(
    source=frame,
    classes=[0],      # Class 0 = person only
    conf=0.5,         # Minimum 50% confidence
    iou=0.45,         # Non-max suppression threshold
    verbose=False
)

for box in results[0].boxes:
    x1, y1, x2, y2 = box.xyxy[0].tolist()
    confidence = box.conf[0].item()
    print(f"Person at ({x1:.0f},{y1:.0f}) conf={confidence:.2f}")

Section 11 — Deep Dive

People Tracking

Members walk in wearing whatever they want. No wristbands, no QR codes, no app required. The system identifies them at the door when they tap their membership card, and then tracks them visually across the gym floor for the rest of their session.

The Full Identification Flow

The system works in three stages: a hard identification at the door, continuous visual tracking on the floor, and a soft confirmation fallback at each machine.

StageWhereHowConfidence
1. Entry IDFront door readerMember taps card / fob100% — hard identity
2. Visual LockEntry cameraYOLO captures appearance at the moment of card tapHigh — known person, known appearance
3. Floor TrackingAll camerasDeepSORT tracks appearance + position continuously95%+ in open space
4. Machine ConfirmMachine tabletTap card again if YOLO confidence drops below threshold100% — hard identity reset

Stage 1 — Card Tap at Entry

When a member taps their card at the door, two things happen simultaneously. The access control system grants entry as normal. At the same moment, the entry camera (a dedicated camera pointed at the door) captures a clean image of the person walking in. YOLO detects the person in that frame and creates a new tracking record: "Tracking ID #42 = John Smith, Member #1089."

This is the most important moment in the entire tracking pipeline. The card tap is the ground truth — we know with 100% certainty who just walked in. Everything after this is the system trying to maintain that certainty as the person moves around the gym.

DeepSORT maintains a persistent tracking ID for each person by combining their predicted position (Kalman filter) with their visual appearance (deep embedding). The card tap at entry is what ties a tracking ID to a real member identity.

Stage 2 — Visual Appearance Lock

At the moment of card tap, YOLO extracts an appearance embedding — a mathematical fingerprint of what that person looks like today. This includes their clothing colours, body shape, and proportions. This embedding is stored against their member ID for the duration of the session.

This is why the system works without face recognition. We don't need to identify their face — we already know who they are from the card tap. The appearance embedding is only used to maintain that identity as they move through the gym, not to establish it.

Stage 3 — Continuous Floor Tracking

As the member moves through the gym, DeepSORT tracks them frame by frame. The tracker uses two signals simultaneously: their predicted position (based on their last known velocity) and their appearance embedding (are they still wearing the same blue shirt?). As long as both signals match, the tracking ID is maintained with high confidence.

Change Room Dead Zone

The change rooms are a deliberate blind spot — no cameras inside. When a member walks into the change room, their tracking ID is suspended. When they emerge, the system attempts to re-identify them by appearance. If they changed clothes, the appearance embedding no longer matches and confidence drops. This triggers Stage 4.

Occlusion Handling

If someone walks behind a machine or another person briefly, DeepSORT holds the tracking ID in a "lost" state for up to 2 seconds. If they reappear in a similar location with a similar appearance, the same ID is restored. If too much time passes, the ID is retired and a new one issued.

Stage 4 — Tablet Confirmation Fallback

Every machine in the gym has a small tablet mounted to it. When a member sits down at a machine, the system checks its confidence that the person at the machine is who it thinks they are. If confidence is above the threshold (~92%), the session is logged automatically with no interruption. If confidence drops below the threshold — because they changed clothes, came from the change room, or look similar to another member — the tablet displays a simple prompt: "Please tap your card to confirm."

This is not an error state — it is a designed feature. The tap takes one second and resets the identity to 100% certainty. The member's workout is never interrupted for more than a moment, and the data integrity is maintained.

Design Principle

The system is designed so that members never need to think about it. They tap their card once at the door (which they already do for access control), work out normally, and occasionally tap at a machine if asked. There are no apps, no wristbands, no QR codes, and no face scanning.

Cross-camera re-identification: when a member moves from one camera zone to another, their appearance embedding is used to link the new tracking ID to their existing member record — no second card tap required.
Section 12 — Technical Deep Dive

YOLO + DeepSORT Integration

Tracking a person through a crowded gym requires two models working in perfect synchronization. YOLO acts as the "eyes" (finding people in a single frame), while DeepSORT acts as the "memory" (connecting those people across time). Here is the technical breakdown of how they integrate, the data handoff, and the specific challenges of a gym environment.

The Architecture & Data Handoff

The integration between YOLO and DeepSORT is a strict, unidirectional pipeline that runs every 33 milliseconds (30 FPS). YOLO does not know about time or history; it only sees the current frame. DeepSORT does not look at raw pixels; it only looks at the bounding boxes YOLO provides.

Gym-Specific Challenges & Solutions

A gym is one of the most hostile environments for computer vision tracking. The integration must be tuned specifically to handle these edge cases.

Challenge 1: Severe Occlusion (The "Squat Rack" Problem)

The Problem: When a person steps inside a squat rack, the vertical steel uprights slice their body into pieces from the camera's perspective. YOLO might detect them as two separate people, or fail to detect them entirely for several frames.

The Solution: We tune DeepSORT's max_age parameter (how long an ID survives without a YOLO detection) from the default 30 frames up to 90 frames (3 seconds). We also lower the YOLO confidence threshold specifically for the "person" class inside designated machine zones. If DeepSORT loses the person behind the rack upright, it holds the ID in memory. When they step out 2 seconds later, the Kalman filter and appearance embedding re-link them instantly without assigning a new ID.

Challenge 2: Similar Appearances (The "Gym Bro Uniform")

The Problem: DeepSORT relies heavily on the appearance embedding (the 128-dim vector) to match people. In a gym, it is extremely common to have three guys wearing identical black t-shirts and black shorts working out next to each other. The appearance vectors will be nearly identical.

The Solution: We shift the matching weight away from appearance and heavily toward spatial distance (Mahalanobis distance). Because our ceiling cameras are looking down at a steep angle, people rarely overlap perfectly in 2D space. By weighting the Kalman filter's spatial prediction at 80% and the appearance embedding at only 20%, DeepSORT correctly tracks the three identical-looking people based on their physical trajectories rather than their clothes.

Challenge 3: Non-Linear Movement (The "Burpee" Problem)

The Problem: DeepSORT's internal Kalman filter assumes linear, constant-velocity movement (like a car driving down a road). Gym movements are explosive, non-linear, and vertical. A person dropping into a burpee completely breaks the Kalman filter's prediction model, causing the tracking ID to detach and float away.

The Solution: We modify the state space of the Kalman filter. Instead of tracking the bounding box center (x, y) and aspect ratio (a, h), we track the bottom-center point of the bounding box (the feet). Because the feet remain relatively stationary on the floor during most exercises (even burpees or squats), the Kalman filter's spatial prediction remains stable while the bounding box expands and contracts above it.

Code Implementation: Tuning for the Gym

To solve the occlusion and dynamic confidence problems discussed above, we wrap the standard YOLO and DeepSORT libraries in a custom pipeline class. This allows us to define "machine zones" where the rules change dynamically.

gym_tracker.py
from ultralytics import YOLO
from deep_sort_realtime.deepsort_tracker import DeepSort

class GymTracker:
    def __init__(self):
        # 1. Initialize YOLOv11 Medium
        self.detector = YOLO('yolov11m.pt')
        
        # 2. Initialize DeepSORT with Gym-Specific Tuning
        self.tracker = DeepSort(
            max_age=90,              # Hold lost IDs for 3 seconds (30fps * 3s)
            n_init=3,                # Require 3 consecutive hits to confirm
            nms_max_overlap=1.0,
            max_cosine_distance=0.4, # Looser appearance matching
            nn_budget=150            # Deep memory bank for appearance vectors
        )
        
        # 3. Define Machine Zones (x1, y1, x2, y2)
        # Inside these zones, occlusion is high (squat racks, cables)
        self.high_occlusion_zones = [
            [100, 200, 400, 800],  # Squat Rack 1
            [800, 150, 1100, 600]  # Cable Crossover
        ]

    def is_in_zone(self, box, zones):
        # Check if the bottom-center (feet) of the box is inside any zone
        x1, y1, x2, y2 = box
        feet_x = (x1 + x2) / 2
        feet_y = y2
        
        for z in zones:
            zx1, zy1, zx2, zy2 = z
            if zx1 < feet_x < zx2 and zy1 < feet_y < zy2:
                return True
        return False

    def process_frame(self, frame):
        # Run YOLO with a very low baseline confidence
        # We will filter the results manually based on location
        raw_results = self.detector(frame, classes=[0], conf=0.20)
        
        detections = []
        for box in raw_results[0].boxes:
            x1, y1, x2, y2 = box.xyxy[0].tolist()
            conf = box.conf[0].item()
            
            # DYNAMIC CONFIDENCE THRESHOLDING
            # If they are in a squat rack, accept a messy 25% confidence detection
            # If they are in open floor space, demand a clean 60% confidence detection
            in_rack = self.is_in_zone([x1, y1, x2, y2], self.high_occlusion_zones)
            
            if in_rack and conf > 0.25:
                detections.append([[x1, y1, x2-x1, y2-y1], conf, 0])
            elif not in_rack and conf > 0.60:
                detections.append([[x1, y1, x2-x1, y2-y1], conf, 0])
                
        # Pass the filtered detections to DeepSORT
        tracks = self.tracker.update_tracks(detections, frame=frame)
        return tracks

Notice how the dynamic threshold works: in open floor space, we demand a high confidence score (60%) from YOLO before we accept the detection. This prevents false positives (like a punching bag being detected as a person). But when a person steps inside the coordinates of the Squat Rack zone, we drop the required confidence to 25%. This means even if YOLO only sees an arm and half a torso through the steel uprights, it still registers the detection, keeping the DeepSORT track alive.

Pose Estimation

Once we know who is in the frame and where they are, we need to understand what their body is doing. Pose estimation maps 17–33 key joints onto each tracked person, giving us a real-time skeleton we can analyse for movement patterns. We start with YOLO Pose and upgrade to a dedicated model in Phase 2 — and critically, this upgrade does not break anything downstream.

The Model Decision: YOLO Pose First

There are several pose estimation models we could use. The key question is: which one do we start with, and what happens when we upgrade? The answer is that the downstream pipeline — the LSTM rep counter, the exercise classifier, the weight detection — only ever sees the 17 keypoint output. It does not care which model produced those keypoints. This means we can swap the pose model at any time without changing anything else in the system.

MoveNet maps 17 body keypoints onto a person performing a barbell squat — shoulders, elbows, wrists, hips, knees, and ankles are all tracked in real time.

The 17 Keypoints

MoveNet outputs 17 (x, y, confidence) coordinates per person, per frame. These cover the full body: nose, left/right eye, left/right ear, left/right shoulder, left/right elbow, left/right wrist, left/right hip, left/right knee, and left/right ankle. From these 17 points, we can calculate any joint angle in the body.

The 17 keypoints MoveNet tracks — numbered 0–16, covering the full body from nose to ankles.

Calculating Joint Angles

The raw keypoints are just pixel coordinates. The real value comes from calculating the angles between joints. For a bicep curl, we calculate the angle at the elbow — the angle formed by the shoulder, elbow, and wrist keypoints. A fully extended arm is ~170 degrees. A fully contracted curl is ~30 degrees. The transition from high to low to high is one rep.

Pose estimation running on multiple people simultaneously — each person gets their own independent skeleton overlay.

Why Camera Angle Matters for Pose

MoveNet was trained primarily on side-on and front-on views of people. A pure top-down (bird's eye) view — where you are looking straight down at someone's head — makes pose estimation unreliable because limbs overlap and the model cannot distinguish left from right. Our ceiling cameras are mounted with a 10–20 degree forward tilt from vertical. This gives a clear view of the body from above-and-slightly-in-front, which is the sweet spot for both pose accuracy and machine coverage.

The fast variant — runs at 50+ FPS on a GPU. Lower accuracy on complex poses. Used in Phase 1 where speed matters more than precision. Runs comfortably on a Mac Mini M4.

The accurate variant — runs at ~30 FPS. Higher accuracy on difficult poses (squats, deadlifts, complex movements). Used in Phase 2+ when we have GPU server capacity.

import tensorflow as tf
import numpy as np
model = tf.saved_model.load('movenet_lightning')
def get_keypoints(person_crop):
    img = tf.image.resize_with_pad(person_crop, 192, 192)
    img = tf.cast(img, dtype=tf.int32)
    outputs = model.signatures['serving_default'](input=tf.expand_dims(img, 0))
    keypoints = outputs['output_0'].numpy()[0][0]  # shape: [17, 3]
    return keypoints  # [y, x, confidence] for each of 17 joints
def calculate_angle(a, b, c):
    # Angle at joint b, formed by points a-b-c
    ba = np.array(a) - np.array(b)
    bc = np.array(c) - np.array(b)
    cosine = np.dot(ba, bc) / (np.linalg.norm(ba) * np.linalg.norm(bc))
    return np.degrees(np.arccos(np.clip(cosine, -1.0, 1.0)))

Rep Counting

A rep is not a single frame — it is a movement pattern that unfolds over time. The LSTM (Long Short-Term Memory) network watches the sequence of joint angles across many frames and learns to recognise the rhythmic pattern of a completed repetition.

Joint angles plotted over time during a set of exercises — each wave is one rep. The LSTM learns to detect these waveforms and count them.

From Keypoints to Rep Count

Every 15 frames (roughly 0.5 seconds at 30 FPS), we take the current joint angles for a tracked person and add them to a rolling window of the last 30 readings — representing the last ~15 seconds of movement. The LSTM processes this sequence and outputs a probability that a rep has just been completed. When that probability crosses a threshold, we increment the rep counter.

The LSTM processes a sequence of joint angle readings over time, learning the temporal pattern of a rep — the down phase, the transition, and the up phase.

Exercise Classification First

Before we can count reps, we need to know what exercise the person is doing. We train a separate classifier that takes the current pose and machine zone and outputs an exercise label: "bench press", "lat pulldown", "bicep curl", etc. This classification determines which joint angles the LSTM should focus on — for a bench press it watches elbow angle; for a squat it watches knee and hip angle.

Elbow angle during a bicep curl set — the regular oscillation between ~170 degrees (extended) and ~30 degrees (contracted) is the rep signal the LSTM learns to count.

Handling Partial Reps and Pauses

Real gym-goers do not perform perfect textbook reps. They pause mid-set, they do partial reps, they rest between sets. The LSTM handles this naturally because it is a sequence model — it only fires when it sees a complete movement pattern, not just any movement. A partial rep does not trigger a count. A pause between sets does not trigger a count. Only a full range-of-motion movement matching the learned pattern counts.

Phase 1 — Pre-trained Models

We start with pre-trained LSTM models from academic research (e.g., the RepCount dataset). These work reasonably well out of the box for common exercises like squats, curls, and bench press.

Phase 3 — Fine-tuned on Our Data

Once we have collected real workout data from the gym, we fine-tune the LSTM on our specific equipment, camera angles, and member movement patterns. This dramatically improves accuracy.

The Hard Problem

The most difficult exercises to count are compound movements like deadlifts and rows, where multiple joints move simultaneously and the movement pattern varies significantly between people. These will be tackled in Phase 2 after the simpler isolation exercises are working reliably.

lstm_rep_counter.py
import torch
import numpy as np
from collections import deque
class RepCounter:
    def __init__(self, window=30):
        self.window = deque(maxlen=window)
        self.model = torch.load('rep_counter_lstm.pt')
        self.rep_count = 0
        self.in_rep = False
    def update(self, joint_angles):
        self.window.append(joint_angles)
        if len(self.window) < 30: return self.rep_count
        seq = torch.tensor(np.array(self.window), dtype=torch.float32)
        prob = self.model(seq.unsqueeze(0)).item()
        if prob > 0.8 and not self.in_rep:
            self.rep_count += 1
            self.in_rep = True
        elif prob < 0.3:
            self.in_rep = False
        return self.rep_count

Edge-to-Cloud Architecture

The MVP relies on a "Dumb Edge / Smart Server" model, streaming heavy video feeds to a central Mac Mini. This does not scale. A production system requires a distributed "Smart Edge / Cloud Brain" architecture to handle hundreds of cameras without saturating local networks.

The Problem with Centralized Inference

Streaming 150 concurrent 4K/1080p video feeds across a gym's local network to a single server creates a massive bandwidth bottleneck. Furthermore, if the central server crashes, the entire gym goes offline. This monolithic approach lacks the fault tolerance required for an enterprise SLA (Service Level Agreement).

// THE ENTERPRISE STANDARD

Major tech companies (AWS, Google, NVIDIA) push inference to the edge. Video is processed as close to the camera as possible. Only lightweight metadata (bounding boxes, keypoints, JSON state) is transmitted over the network.

Distributed Edge AI

In Phase 3, we transition to edge nodes capable of running lightweight inference (e.g., NVIDIA Jetson Orin Nanos or cameras with built-in AI accelerators). The edge nodes run YOLO and MoveNet locally.

The new model runs silently in the background on live data, but its outputs are not shown to the user or saved to the database. To mitigate risk, new software versions (both AI models and API services) are deployed using a Canary strategy. The update is pushed to 5% of gyms first. The observability stack monitors these gyms for increased error rates or latency spikes. If the metrics remain stable for 24 hours, the rollout expands to the rest of the fleet.

Three-Tier Architecture

The system is built across three tiers, each with a distinct role. This separation is not arbitrary — it mirrors how AWS, Google, and NVIDIA structure production edge AI systems. Each tier does exactly one job, and the interfaces between them are clean and replaceable.

// THE THREE TIERS
Tier 1 — Edge (Raspberry Pi): Real-time ONNX inference. Knows nothing about training. Runs three models, applies the State Gate, and uploads uncertain clips to GitHub.

Tier 2 — Brain (Mac Mini): PyTorch training, FastAPI server, PostgreSQL. Knows everything about the data. Exports ONNX models and pushes them to GitHub Releases.

Tier 3 — Hub (GitHub): Immutable data store. Training data, model versions, uncertain clips, and the OTA update pipeline all live here.

Why ONNX on the Pi — Not PyTorch?

PyTorch is the right tool for training — it has autograd, flexible model definitions, and the best research ecosystem. But PyTorch is not the right tool for inference on a Raspberry Pi. The full PyTorch runtime is ~500MB and carries significant CPU overhead. ONNX Runtime is ~30MB, runs 2–3x faster on ARM processors, and has no Python dependency at all if needed. This is the same reason AWS Greengrass, NVIDIA Metropolis, and Google Coral all use ONNX or TFLite for edge deployment. The workflow is: train in PyTorch → export to ONNX → deploy to Pi.

The State Gate

This is the most important piece of logic in the entire system. The State Gate solves a fundamental problem with multi-class neural network outputs: a softmax classifier can produce high confidence scores for multiple classes simultaneously. Without the gate, the system can believe a person is both ENGAGED (mid-rep) and RESTING at the same time — causing double-counted reps, broken session boundaries, and corrupted workout logs.

The Problem Without a Gate

Imagine the model outputs: ENGAGED=0.62, RESTING=0.58, IDLE=0.12. Both ENGAGED and RESTING exceed a naive 0.5 threshold. If both fire, the system logs a rep AND ends the set simultaneously. This is not a hypothetical edge case — it happens constantly at the transition between states, which is exactly when accurate detection matters most. Your instinct to add a gate is correct, and it is the standard approach used in production state machine systems.

The Three Gate Rules

The gate applies three rules in sequence. Rule 1 is checked first and overrides everything else if it fires.

// GATE RULE 1 — UNCERTAIN THRESHOLD
If the maximum confidence score across all 8 states is below 0.50, emit UNCERTAIN regardless of all other scores. The model is not confident enough to make any decision. Upload the clip for human review. This is the safety valve — when in doubt, ask a human.
// GATE RULE 2 — MUTEX SUPPRESSION
Certain state pairs are mutually exclusive by definition. A person cannot be mid-rep and resting simultaneously. If both states in a mutex pair exceed the confidence threshold (0.65), the lower-priority state is suppressed:

• ENGAGED suppresses RESTING — cannot be mid-rep and resting at the same time
• APPROACHING suppresses LEAVING — cannot be moving toward and away simultaneously
// GATE RULE 3 — PRIORITY SELECTION
After mutex suppression, if multiple states remain above the confidence threshold, the highest-priority state wins. Priority is defined by the integer value of the State enum — lower number = higher priority. ENGAGED (priority 1) always beats RESTING (priority 3). The gate returns exactly one state.

The Finite State Machine — Valid Transitions

The gate also enforces valid state transitions. A person cannot jump from IDLE directly to ENGAGED — they must pass through APPROACHING and SETUP first. If the model tries to emit an impossible transition, the gate finds the next valid state or holds the current state. This prevents impossible state jumps caused by brief model errors or camera occlusion.

# edge/state_gate.py

class State(IntEnum):
    UNCERTAIN   = 0  # Always wins — triggers clip upload
    ENGAGED     = 1  # Active rep in progress
    SETUP       = 2  # Adjusting seat/weight
    RESTING     = 3  # Between sets
    APPROACHING = 4  # Moving toward machine
    LEAVING     = 5  # Moving away
    BLOCKED     = 6  # Occupied, not using
    IDLE        = 7  # No person

MUTEX_PAIRS = [
    (State.ENGAGED,     State.RESTING),
    (State.APPROACHING, State.LEAVING),
]

CONFIDENCE_THRESHOLD = 0.65
UNCERTAIN_THRESHOLD  = 0.50

def apply_gate(softmax_output: list[float]) -> GateOutput:
    scores = {State(i): v for i, v in enumerate(softmax_output)}

    # Rule 1: Max confidence below uncertain threshold
    if max(softmax_output) < UNCERTAIN_THRESHOLD:
        return GateOutput(state=State.UNCERTAIN, upload_clip=True)

    # Rule 2: Collect states above confidence threshold
    active = {s: v for s, v in scores.items() if v >= CONFIDENCE_THRESHOLD}

    # Rule 3: Apply mutex suppression
    for (high, low) in MUTEX_PAIRS:
        if high in active and low in active:
            del active[low]  # High priority suppresses low

    # Rule 4: Return highest-priority active state
    return min(active.keys(), key=lambda s: s.value)

The Three Models

Three separate neural networks run in parallel on each Raspberry Pi. They are deliberately separate — not one large multi-task model. This is the correct production approach: each model can be trained, updated, and replaced independently without affecting the others. If the weight detection model needs retraining because a new machine type was added, only WeightNet is retrained — PoseNet and FaceNet are untouched.

Model 1 — PoseNet (The State Classifier)

PoseNet is an LSTM that takes a sequence of 30 frames (approximately 1 second at 30 FPS) and classifies the person's activity into one of 8 states. A single frame cannot distinguish ENGAGED from RESTING — both look like a person sitting at a machine. The LSTM sees the motion pattern over time and detects the characteristic joint angle trajectory of a rep in progress versus a person sitting still. The input is 17 body keypoints (x, y coordinates each) = 34 features per frame, giving an input tensor of shape [batch=1, seq=30, features=34].

Model 2 — FaceNet (Member Identity)

FaceNet uses transfer learning from a pretrained MobileNetV2 backbone to produce a 128-dimensional embedding vector for any face crop. This embedding is compared against stored member embeddings in PostgreSQL using cosine similarity. If the similarity score exceeds 0.85, the member is identified. Below 0.85, the tablet prompts for RFID or manual check-in. The 0.85 threshold is configurable per gym — a higher-security installation might set 0.92.

Model 3 — WeightNet (Weight Detection)

WeightNet reads the current weight setting from the machine. It uses the AlphaFit colour band system as the primary signal — each weight increment has a distinct colour stripe on the weight horn, visible from the overhead camera. WeightNet is a regression model that outputs a float value in kg. For machines without colour bands, it falls back to OCR on the weight stack numbers. The colour band approach is more reliable than OCR because it works at low resolution and in variable lighting.

// WHY THREE SEPARATE MODELS, NOT ONE
A single multi-task model outputting state + identity + weight simultaneously would be harder to train, harder to debug, and impossible to update partially. Three separate models means three independent update cycles, three independent confidence thresholds, and three independent failure modes. This is standard practice at Google (separate models for detection, classification, and embedding) and NVIDIA (separate models in the DeepStream pipeline). The Mac Mini trains each model independently and exports each to its own ONNX file.

OTA Update Pipeline

The Over-The-Air (OTA) update pipeline is what transforms this from a static system into a self-improving one. Every time a Pi encounters a situation it is not confident about, it uploads the clip to GitHub. A human reviews and annotates it. That annotation feeds back into the training data. The model gets retrained and pushed back to all Pis automatically. The system gets smarter with every uncertain moment it encounters — this is called active learning.

Hot-Swap Without Restart

When a Pi downloads a new model, it does not restart. The model_updater.py module downloads the new ONNX to a temp file, validates the SHA256 checksum, then signals the inference thread to reload its ONNX Runtime session. The old session is released and the new one takes over in under 200ms — no downtime, no missed frames, no interrupted sessions. This is the same hot-swap pattern used by NVIDIA Metropolis for edge model updates.

Mac Mini as a GitHub Actions Self-Hosted Runner

The retraining step runs on the Mac Mini, not on GitHub's cloud runners. PyTorch training on an M4 chip (with Metal Performance Shaders acceleration) is significantly faster than on a generic cloud runner, and it keeps all training data local. The Mac Mini is registered as a self-hosted runner in the GitHub Actions settings — a one-time 5-minute setup. The retrain.yml workflow uses runs-on: self-hosted to target it specifically.
